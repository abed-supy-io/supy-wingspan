## supy-configmaps — Kubernetes ConfigMaps (YAML), per-service/per-environment — ⚠️ SECURITY-BLOCKING

- **Purpose:** Central K8s ConfigMap repo for Supy microservices/infra; non-secret env config per service and environment (dev/preview/prod/infra).
- **Structure:** flat root with ~20 per-service dev configmaps (`{service}-dev.yaml`); subdirs `preview-configmaps/`, `prod-configmaps/`, `infra-configmaps/`. Names `{service}-{env}`, namespace `{env}`.
- **Architecture & patterns:** one file per service (no Kustomize/Helm); env stratification via file paths; static K8s DNS SRV service discovery (e.g. NATS cluster names); cloudflared tunnel config stored as ConfigMap multiline strings.
- **Tooling:** validate=**none** · CI=**none** · lint=none · pre-commit=**none** · commits=conventional · deploy=git push (assumed ArgoCD/kubectl watch).
- **Testing:** none (no kubeval/kube-linter).
- **Security / secrets / config:** ⚠️ **CRITICAL — plaintext secrets committed in ConfigMaps.** Affected files (paths only; values NOT reproduced): `supy-api-dev.yaml` (Twilio SID/token, Firebase, Intercom, JSReport, Unleash, Referral Hero tokens), `supy-integration-inventory-dev.yaml` (Lightspeed secrets, Xero token, Tissl API key), `supy-mailgun-webhooks-dev.yaml` (SFTP passwords, SendGrid API key). No config/secret separation; Slack webhooks + OAuth secrets in `data:` blocks.
- **Divergences vs typical k8s config practice:** secrets stored in ConfigMaps (should be `kind: Secret`); no encryption/RBAC separation; no Kustomize/Helm (env duplication); dev tokens/OTPs in plaintext.
- **New patterns worth codifying (as MANDATORY infra rules):** (1) move all tokens/keys/passwords/webhooks to K8s Secrets or external-secrets/sealed-secrets/Vault; (2) split per service into `{svc}-{env}-config.yaml` (non-sensitive) + a referenced Secret; (3) Kustomize bases+overlays to DRY dev/preview/prod; (4) **pre-commit secret-scanning hook that rejects commits**; (5) document ConfigMap-vs-Secret policy; (6) kubeval/kube-linter CI.
- **Recommendation:** **new infra/config standard (BLOCKING).** Secrets MUST be externalized to `kind: Secret` / external store before prod promotion; backfill existing dev/preview configs; add secret-scanning pre-commit + kubeval CI. This directly reinforces the org rule against exposed secrets and is the highest-priority cross-cutting finding of the whole analysis.
