# Skill trigger-overlap registry

Curated registry of every skill-trigger-overlap pair flagged by the "Overlap flag" column of
`docs/skill-audit.md` (R0 Task 1). `scripts/check-skill-hygiene.sh` (Check O) fails CI if any
row below has an empty `Reason` cell, and fails if this file is missing entirely. This registry
documents *known* overlaps and why both skills exist; it does not — and cannot, being a curated
list rather than fuzzy auto-clustering — prove the *absence* of undocumented overlap.

None of the pairs below were merged: each is a deliberate companion or sibling design, already
cross-referenced from both skills' own bodies, not a redundant duplicate (same trigger, same
procedure). Per the R0 resolution, only genuine duplicates would be merged, and a merge would
be recorded here plus in `docs/skill-audit.md` with its replacement trigger noted.

| Skill A | Skill B | Overlap | Reason (why both exist) |
|---|---|---|---|
| supy-scaffold-feature | supy-angular-feature | Both can trigger on "add/build a new Angular feature" in an angular-nx supy repo. | Companions for two different phases: `supy-scaffold-feature` generates the 13+ file NGXS skeleton via the Plop generator (never hand-create it); `supy-angular-feature` is the how-to for the Supy conventions (OnPush, `inject()`, signal inputs/outputs, NGXS+Immer) used to fill that skeleton in. Each cross-references the other (`skills/supy-angular-feature/SKILL.md:184`, `skills/supy-scaffold-feature/SKILL.md:17`). No merge — collapsing them would force the scaffold step and the fill-in how-to into one oversized skill and lose the "scaffold first, never hand-create" gate that `supy-scaffold-feature` enforces on its own. |
| supy-scaffold-domain | supy-clean-architecture | Both can trigger on "add a new domain/bounded-context" in a nestjs-nx supy repo. | Companions for two different phases: `supy-scaffold-domain` generates the 25+ file, five-library bounded-context skeleton via the Plop `g:domain` generator; `supy-clean-architecture` is the how-to for Clean/Hexagonal + DDD + CQRS conventions (aggregates, value objects, factories, interactors) used to fill it in. Each cross-references the other (`skills/supy-clean-architecture/SKILL.md:184`, `skills/supy-scaffold-domain/SKILL.md:17`). No merge — same rationale as the Angular pair above. |
| supy-scaffold-flutter-feature | supy-flutter-feature | Both can trigger on "add a new feature" in a flutter supy repo. | Companions for two different phases: `supy-scaffold-flutter-feature` generates the domain/data/presentation skeleton from bundled stubs and prints the manual wiring; `supy-flutter-feature` is the how-to for Supy Flutter conventions (Clean Architecture layers, Bloc not Cubit, go_router, get_it, dartz Either/Failure) used to fill it in. Each cross-references the other (`skills/supy-flutter-feature/SKILL.md:196`, `skills/supy-scaffold-flutter-feature/SKILL.md:45`). No merge — same rationale as the two pairs above. |
| supy-impl-spec | supy-spike-spec | Both trigger on "write a spec for this Jira ticket" — the distinguishing signal (buildable feature vs. research/investigation) is in the request, not the skill name. | Siblings producing the same `docs/specs/` output shape for two different ticket types: `supy-impl-spec` is for a ticket ready to be built (architecture, testing strategy, implementation details); `supy-spike-spec` is for a ticket that needs investigation first (research questions, options, PoC scope, success criteria). Each cross-references the other under its own "Related skills" section (`skills/supy-impl-spec/SKILL.md:162-168`, `skills/supy-spike-spec/SKILL.md:164-168`) and `supy-spike-spec` documents the hand-off: once a spike lands on a recommendation, it feeds `supy-impl-spec`. No merge — the two specs have different required sections and a single skill would have to branch internally on ticket type, which the router's own trigger-cue mechanism already does more reliably. |
