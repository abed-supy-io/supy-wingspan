# Standards Index

Extracted from real Supy repos on **2026-07-15**. Each file documents rules that the review subagents cite by H2 anchor (`## Rules`, `## DDD building blocks`, `## Examples`, `## Red flags`, `## Source`). Standards are organized by stack: backend (nestjs-nx) files live at the root of this directory (plus [`backend/`](backend/) for module boundaries), frontend files under [`frontend/`](frontend/), mobile (Flutter) files under [`flutter/`](flutter/), the standalone Firebase Functions backend under [`firebase-functions/`](firebase-functions/), the standalone commander.js CLI under [`ts-cli/`](ts-cli/), and the polyglot AI-agents monorepo under [`ai-agents/`](ai-agents/). Three files at the root are **cross-cutting / stack-agnostic** and govern every repo: `commit-conventions.md`, `secrets-and-config.md`, and `ci-coverage-baseline.md` (see the Cross-cutting section below).

**Rule: Refresh these files when the mined source repos change** (new commitlint config, refactored NATS conventions, updated Cerbos policies, Angular/NGXS convention changes, Flutter/BLoC convention changes, stack version bumps).

**Profiles are not stack variants — they are the same stack with two error-transport conventions plus two library shapes.** The reviewer runs a **Step 0** first: `melos.yaml` → melos-packages sub-profile (M1–M5); `flutter.plugin.platforms` in `pubspec.yaml` → plugin sub-profile (P1–P6, no BLoC/go_router/get_it expected); otherwise it is an app, and the `dartz` dependency selects **Profile A** (present) vs **Profile B** (absent). A single app mixing both error-transport profiles is a red flag. The P1–P6 and M1–M5 rules live under `architecture.md#profiles`; findings against them cite that anchor plus the rule tag (e.g. `architecture.md#profiles rule P1`).

## Firebase Functions files (firebase-functions)

The standalone, **non-Nx** Firebase Functions backend (`supy-firebase-functions`) — Node/TypeScript on the Firebase Functions runtime (Node 22), Awilix DI (not NestJS DI), Cloud Firestore, package under `functions/`.

| File | What it covers | Confidence |
|---|---|---|
| [firebase-functions/architecture.md](firebase-functions/architecture.md) | Clean Architecture for Firebase Functions (`index.ts` → app interactors → data repositories → frameworks; dependencies inward), Awilix resolve-don't-`new` + `createScope`, **runtime-enforced auth markers** (`@unauthenticated`/`@internal`/`@admin`/`@apiKey` — a marked handler that never calls its guard is an open endpoint), typed domain errors (`CustomError`→`HttpsError`), **at-least-once Firestore/PubSub trigger idempotency**, secrets from Secret Manager (never literals). Split into per-diff-enforceable `## Rules` (9 numbered) and a repo-wide `## Remediation backlog` (migrations — **not** per-diff blockers unless the diff introduces the defect) | medium |

This is the **least-governed** TypeScript repo at Supy (TSLint not ESLint, no tests, no CI, no pre-commit, unenforced auth markers, historically hardcoded credentials) — the standard is deliberately **remediation-first**. The `supy-firebase-functions-reviewer` agent and the `supy-firebase-function` skill cite `firebase-functions/architecture.md` by H2 anchor (e.g. `architecture.md#rules rule 3`, `architecture.md#red-flags`), and also lean on the cross-cutting `secrets-and-config.md`. The reviewer must **not** raise `## Remediation backlog` items against an unrelated diff. The [`templates/firebase-functions/`](../../templates/firebase-functions/) scaffold supplies the runtime auth guards, an ESLint (TSLint→ESLint migration) config, a CI workflow, a gitleaks/lint-staged/commitlint pre-commit chain, and a stack-specific `CLAUDE.md.hbs`.

## ts-cli files (ts-cli)

The standalone, **non-Nx** commander.js CLI (`supy-cli`) — a MongoDB scripts runner shipped as an npm bin (Node/TypeScript, Clean Architecture, Mongoose per-DB pooling, `inquirer`/`chalk` UX). A single command can mutate a **production** database in bulk, so operational safety is as load-bearing as the architecture.

| File | What it covers | Confidence |
|---|---|---|
| [ts-cli/architecture.md](ts-cli/architecture.md) | Clean Architecture for a CLI (`presentation` commands → `application` scripts/use cases → `infrastructure` → `domain` at the core; dependencies inward, `@domain`/`@application`/`@infrastructure` aliases), every script implements `IScript` with a complete self-documenting `ScriptDetails` block (powers `scripts info`), thin commands that parse/validate/resolve/delegate, layered-env config (never literals), **production-mutation confirmation gates** (explicit interactive confirm naming the target DB; refuse in non-interactive/CI), **no secrets in argv or logs**, deterministic exit codes, unit-testable use cases (DB via context), batched bulk ops on a pooled connection. Split into per-diff-enforceable `## Rules` (9 numbered) and a repo-wide `## Remediation backlog` (add tests/CI/pre-commit, ESLint 8→9, `.env.example` — **not** per-diff blockers unless the diff introduces the defect) | medium |

Like `firebase-functions`, this repo has **no tests, no CI, and no pre-commit** today and runs on ESLint 8 — the standard is deliberately **remediation-first**. The `supy-ts-cli-reviewer` agent and the `supy-ts-cli` skill cite `ts-cli/architecture.md` by H2 anchor (e.g. `architecture.md#rules rule 5`, `architecture.md#red-flags`), and also lean on the cross-cutting `secrets-and-config.md`. The reviewer must **not** raise `## Remediation backlog` items against an unrelated diff. The [`templates/ts-cli/`](../../templates/ts-cli/) scaffold supplies a flat ESLint 9 config (with a hardcoded-secret guard), a CI workflow, a gitleaks/lint-staged/commitlint pre-commit chain, an `.env.example`, and a stack-specific `CLAUDE.md.hbs`.

## ai-agents files (ai-agents)

The polyglot AI-agents monorepo (`supy-ai-agents`) — Node.js + Python + Cloudflare Workers packages (Cortex, Nexus, Oculus, Gleap, PMS-AI, plus a git submodule) with **no root workspace orchestration**; each package is owned and shipped independently. Several packages expose MCP tools / HTTP routes, run BullMQ background jobs, hold OAuth/JWT secrets, and ship in containers, so **secret hygiene, auth on every exposed surface, and idempotent job consumers are as load-bearing as the layering**. (This repo also *hosts* the Cortex MCP.)

| File | What it covers | Confidence |
|---|---|---|
| [ai-agents/architecture.md](ai-agents/architecture.md) | Architecture + operational safety for a polyglot AI-agents monorepo: secrets from env/secret store (never literals/argv/logs; new keys → `.env.example`/`wrangler` placeholders only; a committed credential is merge-blocking); auth (OAuth 2.1/JWT/Bearer) on every exposed MCP tool/resource/route on a mutating or data-returning path; config through an env-driven config singleton (not scattered `process.env`/`os.environ`); every external entrypoint validates input + handles errors + rate-limits; Redis/BullMQ via shared cluster config with fallback; **idempotent queue consumers** (deterministic upsert keys, never logging secret-bearing payloads or cloned-repo content); **non-root containers** (UID 1001); each package self-contained (no `../../<sibling>/` reach-through imports); Conventional Commits + manual semver. Split into per-diff-enforceable `## Rules` (9 numbered) and a repo-wide `## Remediation backlog` (uniform lint/tests/CI/pre-commit across packages — **not** per-diff blockers unless the diff introduces the defect) | low |

This is a **new-signal, low-confidence** stack (grounded in the initial `docs/analysis/supy-ai-agents.md` sweep, not a mature convention corpus) — the standard is deliberately **remediation-first** and its confidence is `low`; refine it as the monorepo's conventions settle. The `supy-ai-agents-reviewer` agent and the `supy-ai-agents` skill cite `ai-agents/architecture.md` by H2 anchor (e.g. `architecture.md#rules rule 2`, `architecture.md#red-flags`), prefer the Cortex MCP for live grounding, and also lean on the cross-cutting `secrets-and-config.md`. The reviewer must **not** raise `## Remediation backlog` items against an unrelated diff. The [`templates/ai-agents/`](../../templates/ai-agents/) scaffold supplies a flat ESLint 9 config (cross-package reach-through + hardcoded-secret guards), a polyglot CI workflow (node / python / secret-scan jobs), a gitleaks/lint-staged (ESLint + Ruff)/commitlint pre-commit chain, an `.env.example`, a non-root `Dockerfile`, and a stack-specific `CLAUDE.md.hbs`.

| File | What it covers | Confidence |
|---|---|---|
| [commit-conventions.md](commit-conventions.md) | Allowed commit types, scope rules, subject/body/footer formatting, `fix` vs `bug` correction (also listed under backend) | high |

| [ci-coverage-baseline.md](ci-coverage-baseline.md) | Minimum automated-verification floor per stack: CI triggers (`pull_request`/`push` to `main` + `cancel-in-progress`), fail-fast check ordering (format→lint→typecheck→test --ci --coverage→build), pinned toolchain versions (backend Node 20 / frontend Node 24 / firebase Node 22 / ts-cli Node 20 / ai-agents Node 20 + Python 3.12 / flutter stable), a **coverage floor enforced as a gate** (flutter app ≥80% / melos ≥85% / plugin ≥70%; Node/TS via Jest `coverageThreshold`), the baseline pre-commit set (**blocking gitleaks first from unit 1**, then `lint-staged` + `nx affected`/`typecheck`), cerbos-compile/kube-linter where the repo holds those artifacts, and the `passWithNoTests` carve-out (shared-library variant only). Remediation-first for the three test-less TS repos except the always-on secret scan | high |

The `supy-secrets-reviewer` agent cites `secrets-and-config.md` by H2 anchor (e.g. `secrets-and-config.md#rules rule 1`, `secrets-and-config.md#red-flags`). It is dispatched by `supy-review` on **every** stack (like `supy-commit-pr-reviewer`), and additionally defines the `k8s-config` stack (ConfigMap/Secret YAML or a `kustomization.yaml`, no app stack). This standard directly enforces the organization rule that secrets must never be exposed — **never reproduce a secret value; cite path:line only.** The [`templates/k8s-config/`](../../templates/k8s-config/) scaffold provides a ready-to-use blocking secret-scan pre-commit config and a config/secret Kustomize split.
